## All Skills by Category

### 1. Product & Requirements Engineering (5 skills)

#### `/grill-me`
**Purpose**: Relentlessly interrogate an idea to extract every detail, assumption, and blind spot  
**When to use**: You have a rough idea, need to flesh it out before building  
**Outputs**: Comprehensive idea document with answers to all critical questions  
**Example**: `/grill-me add real-time notifications to dashboard`

#### `/write-a-prd` ✨ NEW
**Purpose**: Transform ideas into structured, implementable PRDs  
**When to use**: After grilling, need a formal Product Requirements Document  
**Outputs**: 
- Problem statement
- Goals & success metrics
- User stories & acceptance criteria
- Technical scope & constraints
- Risks & mitigation
- Design references
**Example**: `/write-a-prd real-time notification system`

#### `/prd-to-issues` ✨ NEW
**Purpose**: Decompose PRD into scoped, implementation-ready Linear/GitHub issues  
**When to use**: PRD is written, ready to start development  
**Outputs**: Prioritized issue list with:
- Clear title and description
- Acceptance criteria
- Dependencies
- Complexity estimate (Small/Medium/Large)
- Labels
**Example**: `/prd-to-issues` then paste or link PRD

#### `/planning`
**Purpose**: Strategic planning and roadmap thinking at multiple scales  
**When to use**: Need to sequence work across sprints or quarters  
**Outputs**: Roadmap, sprint plans, capacity analysis  
**Example**: `/planning what should we build next quarter?`

#### `/shaping`
**Purpose**: Shape rough ideas into meaningful, scoped, time-bounded work  
**When to use**: Ideas are too vague or too big  
**Outputs**: Shaped work ready for `/grill-me` or `/write-a-prd`  
**Example**: `/shaping how should we approach the dashboard redesign?`

### 2. Development & Testing (4 skills)

#### `/tdd` ✨ NEW
**Purpose**: Apply strict Test-Driven Development: Red → Green → Refactor  
**When to use**: Building new features, want confidence in correctness  
**Process**:
1. RED: Write failing tests (specify behavior)
2. GREEN: Minimal code to pass tests
3. REFACTOR: Improve while keeping tests green
**Outputs**: Tested, well-refactored code  
**Example**: `/tdd implement rate limiter for API endpoints`

#### `/test-writer`
**Purpose**: Generate comprehensive test suites for functions/modules  
**When to use**: Need thorough test coverage, want to catch edge cases  
**Outputs**: Tests for happy path, edge cases, error cases  
**Languages**: Rust, TypeScript, Python, Go, etc.  
**Example**: `/test-writer tests for src/utils/auth.ts`

#### `/code-review`
**Purpose**: Professional code review with specific, actionable feedback  
**When to use**: Before pushing to GitHub, want to self-review  
**Outputs**: Issues, improvements, style notes  
**Example**: `/code-review src/api/handlers/users.ts`

#### `/codex-review`
**Purpose**: Codex/Codex AI-specific code review patterns  
**When to use**: Reviewing code generated by AI agents  
**Focus**: Agent patterns, MCP integration, automation logic  
**Example**: `/codex-review .codex/agents.md`

### 3. Code Improvement & Refactoring (3 skills)

#### `/improve-my-codebase` ✨ NEW
**Purpose**: Systematic codebase analysis and prioritized improvement recommendations  
**Analysis dimensions**:
- Code Quality (duplication, complexity, naming)
- Testing (coverage gaps, edge cases)
- Architecture (coupling, cohesion, SOLID)
- Performance (N+1, memory, caching)
- Security (validation, auth, secrets)
- Maintainability (type safety, docs)
- Operations (logging, monitoring, deployment)
**Outputs**: Prioritized improvement plan with effort/impact  
**Example**: `/improve-my-codebase src/api` (scan and suggest)  
**Example**: `/improve-my-codebase database performance` (focus area)

#### `/debugging`
**Purpose**: Structured debugging approach and systematic issue investigation  
**When to use**: Bug found, need to reproduce and understand  
**Process**: Hypothesis testing, isolation, root cause analysis  
**Outputs**: Reproduction steps, root cause, fix verification  
**Example**: `/debugging why is user dashboard loading slowly?`

#### `/regression-investigator`
**Purpose**: Find and fix regression issues (features that broke after a change)  
**When to use**: Something worked, now it doesn't  
**Process**: Identify what changed, isolate regression, fix + test  
**Outputs**: Root cause, fix, test to prevent recurrence  
**Example**: `/regression-investigator authentication broke after update`

#### `/slidev`

**When to use**: Need to present technical content or create talks  
**Outputs**: Interactive, code-rich presentations  
**Note**: Symlink to shared skills directory  
**Example**: `/slidev create a talk on RF receiver architecture`

#### `/frontend-slides` ✨ NEW
**Purpose**: Create beautiful, animated HTML presentations from scratch or PowerPoint  
**When to use**: Need visually stunning web presentations without CSS/JS knowledge  
**Key features**:
- Zero dependencies (single HTML files)
- Visual style discovery (pick from generated previews)
- PowerPoint conversion (preserves images & content)
- Anti-AI-slop aesthetic (curated distinctive styles)
- Production-ready, responsive, accessible code
**Workflow**:
1. Describe content and desired feeling
2. See 3 visual style previews
3. Pick your favorite style
4. Get full presentation in your chosen style
5. Open in browser
**Conversions**: Also converts `.pptx` files to web slideshows  
**Example**: `/frontend-slides I want to create a pitch deck for my AI startup`  
**Example**: `/frontend-slides Convert my presentation.pptx to a web slideshow`

## Recommended Workflows

### Product Workflow (Idea → Implementation)
```
1. /grill-me [your idea]
   ↓
2. /write-a-prd [create formal document]
   ↓
3. /prd-to-issues [decompose into tasks]
   ↓
4. /planning [sequence across sprints]
   ↓
5. /tdd [implement each issue]
```

### Development Workflow (Code Quality)
```
1. /tdd [new feature - test first]
   ↓
2. /test-writer [ensure complete coverage]
   ↓
3. /improve-my-codebase [refactor & optimize]
   ↓
4. /code-review [self-review before PR]
   ↓
5. /git-push-pr [submit with good description]
```

### Codebase Health Workflow (Continuous Improvement)
```
1. /improve-my-codebase [analyze]
   ↓
2. /planning [prioritize improvements]
   ↓
3. /tdd [implement improvements with tests]
   ↓
4. /git-push-pr [submit improvements]
```

### Bug Investigation Workflow
```
1. /debugging [reproduce & understand]
   ↓
2. /regression-investigator [find what broke]
   ↓
3. /tdd [add test for fix]
   ↓
4. /git-push-pr [submit fix]
```

### Interview Prep Workflow
```
1. /interview [prep for specific role]
   ↓
2. /planning [structured study plan]
   ↓
3. /tdd [practice problem solutions]
```
